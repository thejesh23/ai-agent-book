# Experiment 5-10: Natural Language Interaction ERP Agent (NL → SQL, artifact pattern)

Automatically convert Chinese natural language queries into SQL, executed by the system to directly present the result table. The core is the **artifact pattern**:
The Agent is only responsible for "generating SQL" as the artifact; the actual data query is handled by the database. **The LLM does not carry data itself** —
saving tokens, avoiding calculation errors by the large model, and returning results for tens of thousands of rows in seconds.

## Data Model (Two Tables)

- `employees`: Employee ID, name, department, level (higher number = higher level), hire date, termination date (NULL = currently employed)
- `salaries`: Employee ID, pay date (one record per month, `YYYY-MM-01`), salary

Data is generated by `seed.py` using a fixed random seed (42), generated relatively based on "today", **fully reproducible**:
Approximately 40 employees across 5 departments/multiple levels, including some who have left; salary is generated monthly based on "hire base + fixed annual raise amount",
with each person having a different raise amount (ensuring unique ranking for question 9); and one currently employed employee intentionally has a month's salary deleted (creating the "arrears" scenario for question 10).

## 10 Auto-answered Questions

1. Average tenure per employee　2. Number of active employees per department　3. Which department has the highest average level
4. How many new hires each department had this year/last year　5. Average salary of Department A from March of the year before last to May of last year
6. Which department (A/B) had a higher average salary last year　7. Average salary per level this year
8. Average salary of the most recent month for employees with tenure of less than one year / one to two years / two to three years
9. Top 10 employees with the largest salary increase from last year to this year　10. Are there any salary arrears (employed in a month but not paid)

Where Department A = R&D Department, Department B = Sales Department (specified in the prompt).

## Running

```bash
pip install -r requirements.txt
cp env.example .env      # Fill in OPENAI_API_KEY
python demo.py           # Equivalent to python demo.py run
```

**Universal OpenRouter fallback**: When `OPENAI_API_KEY` is not configured, setting `OPENROUTER_API_KEY` will automatically
switch to OpenRouter (`gpt-*` → `openai/*`, others → `openai/gpt-5.6-luna`). The default model
`gpt-5.6-luna` belongs to gpt-5.x; direct connection to OpenAI requires organizational real-name authentication, so when `OPENROUTER_API_KEY` is set,
it will prioritize OpenRouter.

`demo.py` provides 4 subcommands (without a subcommand, it is equivalent to `run`):

| Subcommand | API Required | Function |
| --- | --- | --- |
| `run` | Yes | Online: Agent generates SQL → executes → compares with reference implementation, prints results per question and gives overall pass rate |
| `gold` | **No** | Offline self-check: Executes built-in "standard SQL" (`gold.py`) for all 10 questions and compares, proving data model consistency |
| `ask` | Yes | Single natural language query → generates SQL → executes and prints result table |
| `initdb` | No | Creates tables and loads seed data into a SQLite file, convenient for manual inspection with `sqlite3` |

Common parameters: `--only 1,5,10` (run only specified question numbers), `--db erp.db` (use file database instead of in-memory database),
`--model gpt-5.6-luna` (override model), `--output result.json` (export per-question details). Examples:

```bash
python demo.py gold                 # Run all 10 questions offline, no API needed
python demo.py run --only 2,3,6     # Only let the Agent generate SQL for these 3 questions and validate
python demo.py ask "How many active employees are in the R&D department now?"
```

Online mode will: Create SQLite in-memory database → load seed data → have the Agent generate SQL for each question → execute → print
"Question / Generated SQL / Query Result / Pass or Fail", and finally give the overall pass rate.

## Correctness Validation

`reference.py` is an **independent Python reference implementation**: It does not use SQL, but calculates the answer for each question directly on the seed data.
`demo.py` compares the SQL execution results with the reference answers using "multiset + numerical tolerance", printing Pass/Fail for each question.
`gold.py` contains 10 manually written "standard SQL" queries. `python demo.py gold` executes them offline for self-checking, no API needed.

Last real run: Offline `gold` pass rate **10/10**; Online `run` (`gpt-5.6-luna`)
all 10 questions passed consistently, overall pass rate **10/10**.

## About the Database

This project uses **SQLite** (zero dependencies, directly reproducible). The book examples use **PostgreSQL**; the SQL is largely compatible,
with differences mainly in date functions: This project uses SQLite's `strftime('%Y','now')`, `julianday()`, `date('now','-1 year')`, etc.;
when migrating to PostgreSQL, replace them with `EXTRACT(YEAR FROM now())`, `AGE()`/date subtraction, `now() - interval '1 year'`, etc.

## Notes and Cautions

- The Agent's prompt includes schema-level hints: expected columns/order, business logic (active = leave_date is NULL,
  how to derive this year/last year using `strftime(...,'now',...)`, Department A/B mapping), and **prohibition of hardcoding years**
  (otherwise the model won't know what year "today" is and will guess "the year before last/last year" incorrectly). These are reasonable schema hints and do not leak specific answers.
- Questions 8 and 10 are more complex (grouping by tenure to get the most recent month's salary, recursively generating employment months to find gaps); recommended
  SQL structure templates are provided in the hints to help smaller models produce correct SQL consistently.
- `temperature=0` makes output as stable as possible, but LLMs are still not strictly deterministic; if individual questions occasionally deviate, re-run,
  or switch to a stronger model (set `OPENAI_MODEL`).
