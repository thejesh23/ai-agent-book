# Afterword: Back to Agent = LLM + Context + Tool {.unnumbered}

This book opened with a formula: **Agent = LLM + Context + Tool**. All ten chapters unfold within these three words.

**Chapter 1** establishes a three-layer understanding of the formula—the implementation layer, the intuition layer, and the academic layer—and presents a spectrum of orchestration from workflows to autonomous agents. Chapters 2 through 7 then build up the three pillars of the formula, one by one:

- **Context (Chapters 2 & 3)—"What the Agent Sees."** Context engineering determines the world the model sees in a single session; user memory and knowledge bases extend this from a single session to long-term accumulation across sessions.
- **Tools (Chapters 4 & 5)—"What the Agent Can Do."** Tools define the boundaries of capability. The Coding Agent is given its own chapter because code is the most universal tool—a tool that can create new tools.
- **Model (Chapters 6 & 7)—"The Agent's Brain Itself."** Evaluation turns the agent's performance into comparable signals; post-training then turns these signals into the model's own capabilities. One measures intelligence, the other amplifies it, and together they determine how far the LLM term in the formula can go.

Chapters 8 through 10 then combine these three pillars and apply them to more complex scenarios:

- **Chapter 8—Self-Evolution.** Enables the agent to accumulate strategies from experience, record workflows, externalize knowledge, and even actively create new tools, all without changing its weights.
- **Chapter 9—Multimodal and Real-Time Interaction.** Extends perception and action from text to vision, speech, and the physical world, confronting the architectural challenges posed by real-time requirements.
- **Chapter 10—Multi-Agent Collaboration.** This is not something new outside the formula: whether context is shared is an expression of "isolation over compression" from Chapter 2 at the system architecture level; "agents as tools for each other" comes directly from the collaborative tool design in Chapter 4; and the "new information" criterion for judging the quality of multi-agent systems echoes the core ideas of evaluation in Chapter 6.

## Two Clouds {.unnumbered}

In 1900, Lord Kelvin said there were two clouds floating over the clear sky of physics—one later became relativity, the other became quantum mechanics. Today, the sky over agents is equally unclear, and I also see two clouds.

**The first cloud is how agents interact with their environment in a streaming, real-time manner.** Today, the vast majority of agents still operate in a turn-by-turn "request-response" mode: you finish a sentence, it thinks through an entire paragraph, and then spits out the result all at once. But the real world doesn't stop to wait for it to finish thinking—speech gets interrupted, visuals are constantly changing, emails keep arriving. A truly "living" agent should be able to listen while thinking, speak while thinking, start planning while you're still in the middle of a sentence, and proactively notice "this email needs handling" even when no one has asked. There are two paths toward this real-time capability, often pursued in parallel: one is **architectural separation of fast and slow**—real-time and intelligence are almost orthogonal axes, hard for a single model to handle both, so a fast front-end model maintains the conversational rhythm while a slow back-end model handles deep thinking; the other is **making inference itself faster**—when decode speed is high enough, the wait time for turn-by-turn becomes so short it nearly disappears, blurring the line between turn-by-turn and "real-time." This path is being rapidly advanced by chips and inference engines: Xiaomi MiMo has pushed a 1T-parameter model past 1000 token/s on a single 8-GPU node[^mimo], while dedicated solutions that hardcode the entire model into a chip (like the Taalas HC1) push an 8-billion-parameter model to about 17000 token/s with a response time under 100 milliseconds[^taalas]. When a model can spit out thousands of words per second, the experiential gap between "think then speak" and "think while speaking" is flattened.

**The second cloud is how agents can continuously accumulate experience from successes and failures in their interactions with the environment, just like humans.** Today's models are more like a genius with an excellent memory who can't learn new things: they memorize human knowledge perfectly during training, but barely grow after deployment—after each task, the pitfalls they've stepped in and the tricks they've discovered are mostly discarded along with the context. Whether this is a real problem depends on two opposing hypotheses.

One is the **"Small World Hypothesis"**: a sufficiently large model—say, with trillions of parameters—already contains almost all the important general knowledge of the physical world; learning once is enough. Those who hold this view (including researchers at OpenAI and Anthropic) would point out that AI is strongest in programming today, not because code is special to models, but because programming is the most open domain for humans: massive amounts of open-source code are available for learning; most industries, however, don't have public information or data. So what frontier labs are really doing is collaborating with various industries one by one, "distilling" their respective professional capabilities into the same large model. According to this view, the bottleneck is neither the model's capacity nor its ability to learn, but whether there's enough data—feed the data in, train once, and the problem is solved.

But the **"Big World Hypothesis"** points to a layer that a single "train once" approach can't cover: knowledge specific to a particular user or a particular company. Your company's coding standards, your team's taste in making PPTs, a certain client's unique temperament—these aren't in any training corpus, and they change constantly. To fit this "big world" made up of countless specific situations, the model can only learn continuously after deployment; it can't be fully equipped at the factory. This is exactly the direction that Chapter 3's memory and Chapter 8's self-evolution are exploring: should experience be written as structured code, stored in a knowledge base, or distilled back into model parameters? Going further, AI has already begun to self-evolve—Anthropic's ongoing talk about "AI for AI" and more and more companies working on "AI for Science" are all about letting agents explore the very frontiers of science; and frontiers have no end, with no ready-made standard answers or corpora to memorize. Agents can only learn autonomously from the successes and failures of their own experiments, rather than turning back to humans for everything. So, the model's strongest capability will ultimately not be memorization, but learning and adaptation.

These two clouds won't be blown away by a single model upgrade. To understand how they will eventually be overcome, we first need to see one thing clearly: models and agents have never been upstream and downstream; they evolve together.

## The Co-Evolution of Models and Agents {.unnumbered}

Looking back at the layers of fallback logic in those harnesses—multi-level context compression, retries that only break after thousands of failures, pessimistic "unsafe" default permission judgments—every piece of seemingly ugly "spaghetti code" records a place where the model is still shaky. When the next generation of models internalizes these constraints, the corresponding code can be deleted; and the reason models can internalize them is precisely because agents have already, in real business scenarios, blazed these trails for them, condensing them into signals for the next round of training. Users pose real challenges, the application layer uses harnesses to compensate for what the model temporarily can't do well, and these compensations in turn become training signals for the model's next iteration—this is a self-reinforcing flywheel, and it's by this flywheel that the two clouds are gradually pushed away.

And the flywheel spins fastest where people hold both ends. What Anthropic is doing with Claude Code is precisely letting its own model and its own harness feed each other and co-evolve: the model knows how the harness will call it, and the harness knows the model's boundaries; every change on either end can immediately feed back to the other. Someone once ran an experiment where, without changing the model, just changing the harness, task accuracy jumped from 52.8% to 66.5%—this shows both how much leverage the harness has today and reminds you: it has this much leverage precisely because the model hasn't gotten there yet. And precisely for this reason, this flywheel itself is the deepest moat of this era: the tighter the coupling between real business, feedback data, and model iteration, the harder it is for others to catch up from the outside.

What this means for you depends on which end of the flywheel you stand on. If you're building models, the moat is to spin this flywheel—let feedback from real-world scenarios flow back into training as quickly as possible. If you're building applications on top of models, the harness is your sharpest short-term technical lever, but be clear-headed: every time the model internalizes a layer of constraints, it will casually flatten a batch of advantages built solely on the harness. The truly lasting moat at the application layer is often outside technology—exclusive data, stable channels, user trust, network effects, and those complex scenarios that AI can't handle in the short term and that require human-agent collaboration. Using the harness to buy time, and using that time to build barriers outside of technology, is the prudent approach.

So, don't worry that the framework you're using will become obsolete. Models iterate every few months; specific APIs, products, and leaderboards will turn over. But the three questions—"what it sees, what it can do, and how to verify if it's doing it right"—will not become obsolete. They describe not the usage of a particular model, but the fundamental way an intelligent system interacts with the world. Master them, and no matter what new capabilities the next generation of models brings, you'll know where to place it in the formula, and you'll be able to see at a glance how far it still is from blowing away those two clouds.

Agent technology is still evolving rapidly; a single book can't keep up with all the changes. But if this book leaves you not with the specific usage of some API, but with a set of judgment skills to stay clear-headed amid the waves of technology, then it has fulfilled its mission. All the main text, illustrations, and accompanying experiment code in this book are open source. You're welcome to go to the repository, run the experiments yourself, and submit issues and PRs—there's a river between understanding and doing that can only be crossed with your own hands. And the most fascinating thing about agents is precisely that they can create new capabilities by writing code, even improving themselves. Having read this far, you already hold the principles of "creation." Now, go create something.
